1. Écosystème global

PokClock est composé de 3 repos applicatifs et de 5 services externes. L'application desktop (PokClock WPF) est la source d'autorité métier, le backend Go (pokclock-api) joue le rôle de bus cloud entre l'app, le site public et les comptes joueurs.

4. Auth license desktop (WPF)

La license POK-XXXX-XXXX-XXXX-XXXX est validée par le Worker Cloudflare (source de vérité), pas par l'API. L'API émet ensuite un JWT RS256 24h utilisé pour les appels suivants.

{ licenseKey, hwid, workerToken } API->>W: POST /verify (forward) W->>D1: SELECT licenses WHERE key = ? AND hwid bound ok D1-->>W: row license W-->>API: { valid: true, tier, role, clubId } API->>API: Signer.Issue(claims) RS256

+ requireClubScope API-->>WPF: structures du club Note over WPF,API: Revalidation /4h via LicenseRevalidationLoop
+ verify synchrone au démarrage (timeout 5s)

5. Auth joueur magic link

Le joueur tape son email sur /joueurs/login, reçoit un lien à 15 min de validité, clique → arrive sur son dashboard. JWT joueur en cookie HttpOnly Domain=.pokclock.com.

6. Auth joueur Google OAuth

Alternative au magic link. Le joueur clique "Continuer avec Google", Google redirige vers l'API, l'API exchange le code et upsert le compte joueur. Pas besoin de re-saisir l'email.

7. Sync Google Calendar (opt-in)

Le joueur connecte son agenda Google via le panel "Mon agenda Google" du dashboard. Chaque inscription tournoi crée automatiquement un event Calendar. OAuth séparé du login avec scope calendar.events + access_type=offline + prompt=consent pour récupérer le refresh_token.

8. Inscription à un tournoi publié

Trois cas selon (audience, joueur authentifié, membership) :

  • Anonyme sur tournoi public : formulaire prénom/nom/email/phone → cancel_token email
  • Joueur authentifié sur tournoi public : OneClick prefilled depuis le profil
  • Joueur authentifié sur tournoi members_only : requiert player_club_memberships.status = 'active'

11. États d'une inscription tournoi

stateDiagram-v2 [*] --> pending : POST register pending --> confirmed : admin POST /confirm pending --> rejected : admin POST /reject pending --> cancelled : joueur DELETE /me/registrations/:id
OU GET /public/registrations/:token/cancel confirmed --> cancelled : joueur ou cancel_token rejected --> [*] cancelled --> [*] confirmed --> [*] : tournoi finished

Transitions interdites : rejected → confirmed, cancelled → confirmed. Pour ré-inscrire un joueur après cancel, l'admin doit lui faire refaire une nouvelle inscription (la précédente reste en trace).

12. États d'une adhésion club (Phase 0.F)

stateDiagram-v2 [*] --> pending : joueur POST /me/clubs/:slug/join pending --> active : admin POST /memberships/:id/approve pending --> rejected : admin POST /memberships/:id/reject active --> revoked : admin POST /memberships/:id/revoke active --> [*] : joueur DELETE /me/clubs/:slug rejected --> pending : re-demande (POST /join réinitialise) revoked --> pending : re-demande

Le re-trigger rejected → pending permet à un joueur de retenter sa demande après refus (peut-être que l'admin avait des questions, le joueur peut re-poster). De même revoked → pending permet la réintégration après exclusion.
