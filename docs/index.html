Backend Go/Echo pour les fonctionnalités cloud de PokClock — license, JWT, sync, audit, backups quotidiens.

Statut Phase 0.A complète · Phase 0.B Étape 3 (backend RBAC) terminée Live api.pokclock.com Repo cedricleblond35/pokclock-api Dernière maj 2026-05-25

1. Vue d'ensemble

Mission

pokclock-api est le backend cloud qui sert l'application PokClock desktop (WPF) en lui apportant trois capacités que le mode standalone ne peut pas offrir :

  • Validation de licence centralisée via un Cloudflare Worker (/verify) et émission d'un JWT RS256 signé.
  • Synchronisation des structures de tournoi (CRUD avec audit log immuable) entre plusieurs postes / clubs.
  • Source d'autorité d'horloge serveur (endpoint /api/time) pour neutraliser la triche par décalage horloge locale.

Pourquoi ce design

  • L'API ne stocke jamais la license en clair — elle délègue la validation au Worker Cloudflare qui détient la source de vérité (D1). L'API ne fait que servir d'intermédiaire entre le client desktop et le Worker, et émet un JWT court (15 min) après validation.
  • Le JWT est signé en RS256 (RSA-2048, clé privée en secret Swarm). La clé publique est exposée via JWKS (/.well-known/jwks.json) → n'importe quel composant peut vérifier un token sans appeler l'API.
  • Postgres ne contient pas de PII — uniquement la métadata métier (structures de tournoi, audit log). Les emails, noms, hardware IDs vivent chez Cloudflare (D1).
  • Backups chiffrés côté client (restic) → Cloudflare R2 n'a jamais accès au contenu en clair.

2. Architecture

2.1 Clean Architecture en couches

Le code suit une organisation Clean Architecture stricte. Les dépendances vont toujours du extérieur vers le centre — jamais l'inverse. Le domaine n'a aucune dépendance sur l'infra.

3.2 Migrations

Les migrations sont gérées par goose v3 et embedées dans le binaire via embed.FS. Elles s'exécutent automatiquement au démarrage de l'API.

VersionFichierDescription
1001_structures.sqlExtension uuid-ossp, table structures + trigger set_updated_at()
2002_structures_audit.sqlTable structures_audit, FK CASCADE sur structures, index (structure_id) + (created_at DESC)
3003_clubs.sqlTable clubs (plan solo/club/pro, status active/suspended/deleted) + indexes status & plan + trigger updated_at
4004_licenses.sqlTable licenses (FK clubs CASCADE, role member/admin/superadmin, worker_token_hash, status active/revoked/expired) + indexes club_id, status, hardware_id partial
5005_structures_club_id.sqlALTER structures ADD club_id (nullable pour rétro-compat Phase 0.A) + index partial
Astuce : pour créer une nouvelle migration, suis la convention NNN_description.sql (3 chiffres, padding zéro) et incrémente. Goose détecte automatiquement la nouvelle version.

3.3 Pool de connexions

Le pool pgxpool.Pool est configuré dans internal/db/postgres.go avec des limites raisonnables pour 2 replicas API + 1 service backup : 

cfg.MaxConns = 10
cfg.MinConns = 2
cfg.MaxConnLifetime = 30 * time.Minute
cfg.MaxConnIdleTime = 5 * time.Minute

Total potentiel : 30 connexions concurrentes côté Postgres (10 × 3 services), bien en dessous du max_connections = 100 par défaut.

4. Cas d'usage

UC1 — Activation license + obtention JWT

Acteur App WPF (poste de croupier) Endpoint POST /api/auth/issue Rate-limit 10/min/IP

Au démarrage, l'app envoie à l'API son triple identifiant (license + hardware + worker token persistés dans license.json après une activation initiale via le Worker). L'API forward la requête au Worker Cloudflare, qui consulte D1 et répond 200 ok si tout matche. L'API émet alors un JWT RS256 valide 15 min.

sequenceDiagram autonumber participant WPF as App WPF participant API as pokclock-api participant W as Worker /verify participant D1 as Cloudflare D1 WPF->>API: POST /api/auth/issue API->>API: rate-limit check 10 per min API->>W: POST /verify with licenseKey hwid token W->>D1: SELECT licenses D1-->>W: row W->>W: HMAC validate workerToken alt valid W-->>API: 200 ok with sub API->>API: signer.Issue sub exp 15m API-->>WPF: 200 token expires_in 900 else invalid W-->>API: 401 bad_signature API-->>WPF: 401 license_invalid end

UC2 — Refresh JWT (sans re-validation Worker)

Acteur App WPF Endpoint POST /api/auth/refresh Auth JWT actuel en Bearer

Pour éviter d'appeler le Worker à chaque fois (et donc surcharger D1 inutilement), le client peut rafraîchir son JWT tant que l'ancien est encore valide. Le nouveau token a même sub et nouvelle expiration. Si le JWT est expiré ou révoqué, il faut repasser par UC1.

sequenceDiagram autonumber participant WPF as App WPF participant API as pokclock-api WPF->>API: POST /api/auth/refresh with Bearer old_jwt API->>API: jwt verify signature and exp alt valid API->>API: signer.Issue same sub new exp 15m API-->>WPF: 200 token expires_in 900 else expired API-->>WPF: 401 token_expired Note over WPF: client doit refaire UC1 end

UC3 — Créer / Modifier une structure de tournoi (avec audit log)

Acteur App WPF (utilisateur authentifié) Endpoints POST /api/structures · PATCH /api/structures/:slug Auth JWT Bearer obligatoire

Chaque création/modification est transactionnelle : la mise à jour de structures et l'écriture de la ligne d'audit dans structures_audit se font dans la même TX. Si quoi que ce soit échoue, rien n'est persisté. L'actor du log = la claim sub du JWT.

sequenceDiagram autonumber participant WPF as App WPF participant API as pokclock-api participant PG as Postgres WPF->>API: PATCH /api/structures/main-clock Bearer jwt API->>API: jwtAuthMiddleware verify and extract sub API->>PG: BEGIN API->>PG: UPDATE structures SET settings API->>PG: INSERT structures_audit row API->>PG: COMMIT PG-->>API: ok API-->>WPF: 200 structure JSON

UC4 — Backup quotidien Postgres → R2

Acteur Service pokclock-api_backup (cron-like) Fréquence Toutes les 24h (sleep 86400) Stockage Cloudflare R2 bucket pokclock-backups

Le service backup tourne en permanence dans une boucle while true. Au démarrage il installe postgresql16-client (l'image restic/restic est alpine mais sans pg_dump), puis initialise le repo restic chiffré sur R2 si pas encore fait, puis enchaîne pg_dump + upload + prune en boucle.

sequenceDiagram autonumber participant Cron as backup loop participant Apk as apk participant Restic as restic participant PG as Postgres participant R2 as Cloudflare R2 Cron->>Apk: apk add postgresql16-client au boot Cron->>Restic: restic snapshots probe alt repo absent Restic->>R2: restic init R2-->>Restic: repo created end loop chaque 24h Cron->>PG: pg_dump Fc f tmp dump PG-->>Cron: dump file Cron->>Restic: restic backup tag pokclock-pg Restic->>R2: upload chunks chiffres dedup R2-->>Restic: snapshot id saved Cron->>Restic: restic forget keep daily 7 weekly 4 monthly 6 prune Restic->>R2: delete expired Cron->>Cron: rm dump and sleep 86400 end

UC6 — Gestion super-admin des clubs (CRUD)

Acteur Cédric (super-admin, JWT role=superadmin) Endpoints GET/POST/PATCH/DELETE /api/admin/clubs[/:id] Auth JWT Bearer + middleware requireRole(superadmin)

Tous les endpoints sont protégés par une double couche : jwtAuthMiddleware (vérifie la signature et l'expiration) puis requireRole(auth.RoleSuperadmin) (vérifie la claim role). Un JWT valide mais avec role=member ou role=admin obtient HTTP 403 insufficient_role.

sequenceDiagram autonumber participant UI as ops.pokclock.com participant API as pokclock-api participant PG as Postgres UI->>API: POST /api/admin/clubs Bearer jwt_superadmin API->>API: jwtAuth verify signature + exp API->>API: requireRole superadmin alt role != superadmin API-->>UI: 403 insufficient_role else role == superadmin API->>PG: INSERT INTO clubs name plan status PG-->>API: row inserted API-->>UI: 201 club JSON end

Opérations supportées sur /api/admin/clubs :

  • GET /api/admin/clubs?status=active&q=paris&limit=50&offset=0 — liste paginée + filtres
  • POST /api/admin/clubs — création ({name, plan, email?, notes?})
  • GET /api/admin/clubs/:id — détail d'un club
  • PATCH /api/admin/clubs/:id — update name/plan/email/notes (status géré séparément)
  • POST /api/admin/clubs/:id/suspend — passer en status=suspended + suspended_at=now()
  • POST /api/admin/clubs/:id/unsuspend — réactiver
  • DELETE /api/admin/clubs/:id — soft delete (status=deleted, ligne conservée pour audit)

UC7 — Émission et gestion des licenses

Acteur Cédric (super-admin) Endpoints GET/POST /api/admin/licenses[/:id/revoke] Auth JWT role=superadmin

Lors de la création, le serveur génère deux secrets : une license_key au format POK-XXXX-XXXX-XXXX-XXXX (base32 lisible) et un worker_token de 32 bytes hex. Seul le SHA-256 hex du worker_token est stocké en base. Le clair n'est jamais persisté côté serveur ; il est renvoyé UNE SEULE FOIS dans la réponse HTTP 201 et doit être copié immédiatement par l'admin (puis communiqué au client final via un canal sécurisé).

sequenceDiagram autonumber participant UI as ops.pokclock.com participant API as pokclock-api participant PG as Postgres UI->>API: POST /api/admin/licenses Bearer jwt_superadmin clubId role API->>API: jwtAuth + requireRole superadmin API->>API: generate license_key POK-XXXX... API->>API: generate worker_token 32 bytes hex API->>API: hash sha256 token API->>PG: INSERT licenses club_id key hash role expires_at PG-->>API: row inserted API-->>UI: 201 license JSON + licenseKey CLAIR + workerToken CLAIR Note over UI: copy now — never shown again

Important pour le futur Worker /verify : quand le Worker recevra (licenseKey, hardwareId, workerToken), il devra hasher le token reçu en SHA-256 hex et comparer au worker_token_hash en base pour valider. Sans cette étape côté Worker, les licenses générées via cet endpoint ne pourront pas s'auth.

Revoke : POST /api/admin/licenses/:id/revoke set status=revoked + revoked_at=now(). Idempotent. Le Worker D1 doit être notifié séparément pour invalider côté Cloudflare (Phase 0.B-2).

UC8 — Recherche dans l'audit log cross-clubs

Acteur Cédric (super-admin) Endpoint GET /api/admin/audit Auth JWT role=superadmin

Recherche full filtres dans structures_audit avec jointure sur structures pour ramener le slug / nom de structure et le club_id de chaque action. Utile pour le support client (« qui a modifié la structure X et quand »).

Filtres disponibles (query params, tous optionnels) :

  • ?clubId=<uuid> — restreint au club donné
  • ?actor=<text> — filtre exact sur l'acteur (claim sub du JWT du croupier)
  • ?action=created|updated|deleted
  • ?from=<RFC3339> et ?to=<RFC3339> — bornes temporelles
  • ?structureId=<uuid> — historique d'une structure précise
  • ?limit&offset — pagination (default 100, max 500)

5. Design Patterns

Les patterns ci-dessous sont ceux effectivement présents dans le code, pas une liste théorique. Chaque ligne pointe vers le fichier exact où ça se passe.

5.1 Patterns Créationnels

Dependency Injection (manual, via struct)
cmd/api/main.go · internal/api/router.go (Deps struct)

Pas de framework DI. Toutes les dépendances (pool DB, signer JWT, client Worker, logger) sont construites dans main.go puis injectées dans api.Mount(echo, Deps{...}). Ça rend les handlers triviaux à tester (on passe un mock dans la struct). 

type Deps struct {
    BuildSHA       string
    Pool           *pgxpool.Pool
    Signer         *auth.Signer
    WorkerClient   *auth.WorkerClient
    AllowedOrigins []string
    Logger         *slog.Logger
}
Factory Method (construction du Signer)
internal/domain/auth/signer.go (NewSigner)

NewSigner(privPath, pubPath) est une factory qui lit les fichiers PEM, parse les clés RSA, calcule le kid (SHA-256 du modulus encodé), et retourne un *Signer prêt à signer + servir JWKS. Le caller n'a jamais à manipuler des *rsa.PrivateKey ou des PEM blocks directement.

Singleton de fait (le pool pgx)
internal/db/postgres.go · cmd/api/main.go

Un seul *pgxpool.Pool est créé au démarrage et partagé entre tous les handlers. Pas un singleton GoF strict (pas de GetInstance() global) — c'est une instance unique par processus portée par l'injection. Bien plus testable qu'un vrai singleton.

5.2 Patterns Structurels

Facade (package internal/api)
internal/api/router.go (api.Mount)

Le package api est une facade au sens GoF : il expose une seule fonction Mount(e, deps) qui configure tout l'aspect HTTP (routes, middlewares, groupes, rate-limit). main.go n'a pas besoin de connaître Echo, les groupes, ou les handlers individuels.

Adapter (pgx stdlib pour goose)
internal/db/postgres.go (RunMigrations)

Goose attend un *sql.DB standard (driver database/sql), mais on utilise pgxpool qui n'expose pas cette interface. L'adapter stdlib.OpenDB(connConfig) de pgx convertit la config en *sql.DB compatible le temps des migrations, puis on referme. Aucun changement côté goose.

Repository (pool injecté dans les handlers)
internal/api/structures.go

Chaque handler reçoit pool *pgxpool.Pool en dépendance et l'utilise pour ses requêtes. Pas d'ORM. Pour Phase 0.B, on pourra extraire un vrai StructuresRepository interface si on veut mocker plus finement, mais à ce stade l'overhead n'est pas justifié.

Decorator (Echo middlewares)
internal/api/router.go · middleware.go

Chaque middleware (Recover, RequestID, slog, CORS, JWT, RateLimit) décore le handler suivant en ajoutant un comportement transverse. Pattern Decorator pur : func(next HandlerFunc) HandlerFunc. Les middlewares peuvent se chaîner sans connaître leurs voisins.

5.3 Patterns Comportementaux

Role-Based Guard (middleware factory)
internal/api/middleware.go (requireRole, requireAnyRole)

Factory de middleware qui retourne un guard sur le rôle JWT. Pattern Factory côté retour, Chain of Responsibility côté usage. Le middleware lit les claims depuis le contexte (déposés par jwtAuthMiddleware en amont), compare au rôle requis, court-circuite avec 403 si insuffisant. Permet d'écrire group.Use(requireRole(auth.RoleSuperadmin)) sans répéter la logique dans chaque handler.

Chain of Responsibility (pipeline Echo)
internal/api/router.go (Mount)

L'ordre des e.Use(...) définit la chaîne. Chaque middleware peut :
1. Court-circuiter (rate-limit dépassé → 429 sans appeler next),
2. Passer au suivant (next(c)),
3. Modifier le contexte après (logger après next).
C'est exactement le pattern Chain of Responsibility.

Strategy (cert resolver Traefik)
pokclock-infra/traefik/docker-compose.swarm.prod.yml

Deux résolveurs ACME coexistent : myresolver (HTTP-01) pour les hosts historiques, ovh-dns (DNS-01 OVH) pour les nouveaux + wildcards. Chaque router Traefik choisit sa stratégie via le label tls.certresolver=.... Stratégies interchangeables sans toucher au code de l'app.

Template Method (entrypoint backup)
docker-compose.swarm.prod.yml (service backup, command)

Le squelette est figé : apk add → restic init/probe → loop(pg_dump → restic backup → forget → sleep). Les "hooks" (creds, repo URL, retention policy) sont injectés via env vars et secrets. Si demain on change Postgres pour MariaDB, on remplace juste l'étape pg_dump par mysqldump, le reste tient.

Observer (slog + RequestID)
internal/api/router.go (slogMiddleware) · middleware.RequestID

Le middleware slogMiddleware observe chaque requête après next() et émet une ligne de log structurée (méthode, path, status, latence, request ID). Les observateurs (Loki, Promtail, ou simplement la stdout JSON) consomment sans coupler le handler à eux.

7. Commandes essentielles

7.1 Développement local

Démarrer l'API en local (sans Docker) Contexte : dev itératif rapide
Lance directement le binaire Go contre une Postgres locale. Tu as besoin d'une instance Postgres qui tourne (Docker ou installation native) et d'un keypair JWT généré localement. 
# Générer un keypair JWT pour le dev
make keypair
# → produit .secrets/jwt_priv.pem et .secrets/jwt_pub.pem (gitignored)

# Lancer Postgres local (si pas déjà)
docker run --rm -d --name pgdev \
  -e POSTGRES_USER=pokclock -e POSTGRES_PASSWORD=dev -e POSTGRES_DB=pokclock \
  -p 5432:5432 postgres:16-alpine

# Exporter les vars d'env minimales
export DATABASE_URL="postgres://pokclock:dev@localhost:5432/pokclock?sslmode=disable"
export JWT_PRIVATE_KEY_PATH=.secrets/jwt_priv.pem
export JWT_PUBLIC_KEY_PATH=.secrets/jwt_pub.pem
export WORKER_VERIFY_URL="http://localhost:8787/verify"  # ou un mock httptest
export ALLOWED_ORIGINS="http://localhost:3000"
export LOG_LEVEL=debug

# Run
make run   # ou : go run ./cmd/api
Sortie attendue : {"time":"...","level":"INFO","msg":"starting pokclock-api","build":"dev","port":"8080"} {"time":"...","level":"INFO","msg":"goose: successfully migrated database to version: 2"} {"time":"...","level":"INFO","msg":"server listening","addr":":8080"}
Tests + couverture Contexte : avant un commit 
make test         # go test -race ./...
make vet          # go vet ./...
make lint         # golangci-lint run ./...
Le -race détecte les race conditions à l'exécution. À toujours utiliser localement avant un push.

8. Particularités & pièges connus

Chaque ligne ci-dessous est un piège qu'on a effectivement rencontré et résolu durant la Phase 0.A. Si tu vois ces symptômes en revenir, le fix est déjà dans le code — ne re-débugge pas de zéro.

Docker env values ne sont PAS passées dans un shell.

Symptôme : parse pgx config: invalid port ":$(cat "

Cause : un compose comme DATABASE_URL: "postgres://user:$(cat /run/secrets/...)@..." n'expanse pas $(cat...). La string est passée littéralement comme env var.

Fix : entrypoint inline dans le Dockerfile qui lit le secret en shell et exporte les bonnes vars. Voir Dockerfile lignes ~35-50 (heredoc RUN cat > /app/entrypoint.sh <<'EOF').

Mot de passe avec caractères spéciaux casse le DSN URL.

Symptôme : invalid port ":<quelque-chose>" after host après que l'entrypoint a lu le secret.

Cause : si le password contient :, /, @, ?, le parseur URL pgx le découpe au mauvais endroit.

Fix : on n'embed PAS le password dans une URL. L'entrypoint exporte PGPASSWORD (var libpq standard, lue nativement par pgx) et construit un DSN au format keyword sans password : host=postgres port=5432 user=pokclock dbname=pokclock sslmode=disable.

L'image restic/restic ne contient pas pg_dump.

Symptôme : container backup exit 127, log /bin/sh: pg_dump: not found.

Cause : l'image upstream restic est minimaliste (alpine + restic).

Fix immédiat : apk add --no-cache postgresql16-client au boot du container (compose service backup, ~3s overhead).

Fix long terme (todo) : construire un Dockerfile.backup dédié qui pré-installe les deux et le pousser sur GHCR.

docker stack deploy ne propage pas toujours les changements d'env vars.

Symptôme : nouveau compose pushé, deploy effectué, mais docker service inspect montre encore les anciennes env vars.

Cause : état Swarm bloqué à cause d'updates antérieurs en --update-failure-action=pause ou rollbacks répétés.

Fix : docker stack rm pokclock-api + sleep 20 + redeploy. Les volumes pokclock_pg_data et tous les secrets external: true survivent.

Cloudflare R2 : éditer un token peut rotater la Secret Access Key.

Symptôme : Stat: Unauthorized en boucle alors qu'on a "juste modifié" les permissions.

Cause : modifier les permissions d'un token User API peut invalider l'ancienne clé secrète selon les conditions Cloudflare. Comportement non déterministe d'après nos tests.

Fix : ne JAMAIS éditer un token R2. Toujours Delete + Create fresh, et bien copier les deux nouvelles clés avant de quitter la page (la Secret Access Key n'est affichée qu'UNE fois).

L'image GHA peut être "cachée stale" lors d'un build.

Symptôme : on ajoute un COPY foo /app/foo dans le Dockerfile, on push, mais l'image résultante sur GHCR ne contient pas /app/foo.

Cause : suspectée du cache type=gha qui n'invalide pas correctement le layer COPY (jamais 100% confirmée).

Fix : embed le fichier inline dans le Dockerfile via RUN cat > /app/foo <<'EOF' ... EOF. Plus de dépendance à un fichier externe = plus de problème de cache.

Le log middleware affichait status=200 pour les 404.

Symptôme : slogMiddleware log 200 sur /wp-login.php etc., alors que le client reçoit bien un 404 (curl le confirme).

Cause : le middleware tourne avant l'HTTPErrorHandler d'Echo, donc res.Status reste à la valeur par défaut (200) quand next() retourne une erreur.

Fix : dériver le status depuis *echo.HTTPError via errors.As. Voir internal/api/router.go commit 0356d40.

zsh : read -p n'existe pas.

Symptôme : read: -p: no coprocess

Fix : printf "prompt: " ; read -s VAR ; echo à la place. Compatible POSIX + zsh + bash.

9.2 JWT keypair

  • Algorithme : RS256 (RSA-2048)
  • Clé privée : jwt_private_pokclock en secret Swarm, jamais en clair sur disque
  • Clé publique : exposée publiquement via /.well-known/jwks.json (kid calculé via SHA-256 du modulus)
  • Backup hors backup : la clé privée n'est PAS dans pg_dump. Si tu perds le secret Swarm, tous les JWT émis deviennent invalides. À sauvegarder séparément (gestionnaire de mots de passe perso, p.ex.) en cas de réinstall complet.

9.3 RBAC — rôles et middleware

Le JWT porte une claim role avec trois valeurs possibles :

RôleAudiencePeut
memberCroupier / utilisateur standardCRUD structures de SON club uniquement
adminAdmin d'un club (président, gérant)Tout ce qu'un member peut + gestion membres + plan + licenses du club
superadminCédric uniquementTout, cross-clubs, accès /api/admin/*

Middlewares dans internal/api/middleware.go :

  • requireRole(role) — exige exactement ce rôle. 401 si non authentifié, 403 si rôle insuffisant.
  • requireAnyRole(roles...) — OU logique pour endpoints multi-rôles (ex: admin OR superadmin).

Toutes les routes /api/admin/* sont protégées par jwtAuth + requireRole(superadmin). Double couche pour défense en profondeur : un JWT valide mais avec un rôle inférieur est rejeté avant d'atteindre le handler.

9.4 Rate limiting

  • /api/auth/issue et /api/auth/refresh : 10 requêtes/min/IP via Echo RateLimiter mémoire
  • Le reste des endpoints n'est pas rate-limité côté API (Traefik peut le faire au niveau réseau si besoin)

9.5 Isolation réseau

  • nw-web : overlay partagé avec Traefik. Seul l'API y est exposé.
  • nw-pokclock : overlay interne au stack. Postgres et backup n'y sont PAS accessibles depuis l'extérieur. Pas de ports: exposé sur l'host.

9.6 CORS

Configuré dans config.go via ALLOWED_ORIGINS. En prod : https://pokclock.com,https://www.pokclock.com. L'app WPF desktop n'envoie pas d'origin → CORS ne s'applique pas, seuls les browsers sont concernés.

9.7 Audit log immuable

Toutes les modifications sur structures écrivent une ligne structures_audit avec : action (created/updated/deleted), diff jsonb, actor (claim sub du JWT), timestamp. La table audit n'a aucune route d'écriture exposée. Lecture en mode super-admin via GET /api/admin/audit (recherche full filtres cross-clubs).

9.8 Génération & stockage des license_keys / worker_tokens

Quand un super-admin crée une license via POST /api/admin/licenses :

  • license_key : format POK-XXXX-XXXX-XXXX-XXXX (4 groupes de 4 chars base32 sans I/O/0/1). Stockée en clair en base — c'est l'identifiant humain.
  • worker_token : 32 bytes crypto/rand, encodés en 64 chars hex. JAMAIS stocké en clair. On stocke uniquement son SHA-256 hex dans licenses.worker_token_hash.
  • Les deux valeurs en clair sont renvoyées UNE SEULE FOIS dans la réponse HTTP 201. Si l'admin ne les copie pas, elles sont perdues — on peut générer une nouvelle license mais pas récupérer celle-ci.
  • Côté Worker Cloudflare /verify (Phase 0.B-2) : pour valider un appel, hasher le workerToken reçu en SHA-256 hex et comparer au worker_token_hash en base. Égalité = OK.

11. Glossaire

TermeDéfinition
JWKSJSON Web Key Set. Format standard pour publier des clés publiques. Endpoint /.well-known/jwks.json.
JWTJSON Web Token. Token signé contenant des claims (sub, exp, iat). Ici en RS256.
RS256RSA Signature with SHA-256. Algorithme asymétrique : signature avec clé privée, vérification avec clé publique.
kidKey ID dans un JWT/JWKS. Permet d'identifier quelle clé a signé le token quand il y en a plusieurs. Calculé via SHA-256 du modulus.
DSNData Source Name. Chaîne de connexion à une DB. Deux formats supportés par pgx : URL (postgres://...) et keyword (host=... user=...).
libpqLib C cliente de Postgres. Définit des conventions (env vars PGHOST, PGPASSWORD, etc.) que pgx respecte.
DNS-01 / HTTP-01Challenges ACME (Let's Encrypt). HTTP-01 = file sur l'host, DNS-01 = record DNS. DNS-01 est requis pour les wildcards.
Swarm secretValeur sensible chiffrée par le raft Swarm, montée en read-only dans /run/secrets/<name> à l'intérieur des containers qui l'utilisent.
Overlay networkRéseau Docker virtuel qui s'étend sur plusieurs nodes Swarm. Permet la communication inter-services.
GHCRGitHub Container Registry. Registry Docker hébergé par GitHub, lié au repo.
resticOutil de backup chiffré, dédoublonné, compatible S3 et bien d'autres backends.
R2Object storage de Cloudflare, compatible S3 API, zero egress fee.
GooseOutil de migrations SQL pour Go. Versions séquentielles, état tracké dans goose_db_version.
pgxDriver Postgres natif pour Go. Plus performant que database/sql+lib/pq, mais expose une API distincte.
