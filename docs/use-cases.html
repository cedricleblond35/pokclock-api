🎲 Dealer / croupier

Le dealer utilise un sous-mode de WPF (license role=member) sur son poste, ou la PWA dealer sur tablette connectée au LAN local du club. Il enregistre les éliminations, demande des resits, déclenche le hand-for-hand.

La PWA dealer est servie par le Kestrel embarqué de WPF (port 5050) en LAN uniquement. SignalR temps réel pousse les états (timer, blinds, joueurs restants) sur toutes les tablettes connectées. Pas d'appel à api.pokclock.com depuis le dealer — tout reste local au club.

🔗 Scénario — Auto-link compte joueur via emailHash (Phase 0.H)

Problème résolu : un club gère ses joueurs habitués dans son app PokClock WPF (saisie manuelle ou import CSV avec emails). Plus tard, ces joueurs créent leur compte cloud sur pokclock.com. Sans Phase 0.H, l'admin doit valider manuellement chaque demande d'adhésion. Avec 0.H, le matching se fait automatiquement par email — l'admin valide juste d'un clic.

Acteurs impliqués

  • Admin du club (TD propriétaire) — saisit les Players dans WPF avec leur email, publie ses tournois
  • Joueur (Jean Dupont, jean@example.com) — habitué du club, joue plusieurs fois par an, n'avait pas encore de compte cloud
  • App WPF — calcule le hash SHA-256 de l'email au moment du publish, l'inclut dans le snapshot
  • Backend pokclock-api — exécute le matching au moment de la création de compte

Variantes & edge cases

SituationComportement
Jean est connu de plusieurs clubs (a joué dans 3 clubs différents)3 demandes pending créées simultanément, une par club. Chaque admin valide indépendamment.
Jean avait déjà une membership active au club X avant 0.HON CONFLICT (player_id, club_id) DO NOTHING → aucune modif. L'active reste active.
Jean a une membership rejected ou revokedIdem : pas écrasé. L'admin avait dit non, on respecte. Jean peut re-demander manuellement via "Rejoindre le club".
Le Player WPF n'a pas d'emailPas d'emailHash calculé → skip silencieux. Jean devra demander l'adhésion manuellement.
Tournoi du club annulé (status=cancelled)Exclu de la recherche. Le matching ne se déclenche pas via un tournoi annulé.
Hash ne matche aucun clubPas de membership créée. Jean a juste un compte cloud "libre", peut s'inscrire en un clic aux tournois publics.

📊 Scénario — Dashboard joueur agrégé par club (Phase 0.I)

Problème résolu : avant 0.I, le dashboard d'un joueur était une liste plate de ses adhésions + une liste plate de ses inscriptions. Pour voir son rang dans un club, il devait naviguer vers la page leaderboard du club, et idem pour vérifier la saison en cours ou les prochains tournois. Avec 0.I, chaque club actif a son propre widget concentré : rang, saison active, prochains tournois avec inscription en un clic.

Narratif

Personae :

  • Sophie : joueuse régulière du club "Poker Club Bordeaux" (rang #4 sur 27 entrées, 312 points), occasionnelle au "Club de la Garonne". Membre active des deux.
  • Le club PCB : a une saison "Printemps 2026" en cours (active), 3 tournois à venir dans les 2 prochaines semaines.
TActionÉtat système
T0 Sophie clique sur "Mon compte" depuis l'app pokclock.com. Le serveur Next.js fait GET /api/players/me/dashboard avec son cookie session.
T1 Le backend liste ses 2 memberships actives, calcule le rang dans chaque club, fetch la saison active, fetch les 3 prochains tournois. Réponse 200 { widgets: [pcb, garonne] }, tri par dernière activité (PCB a un tournoi demain, Garonne le mois prochain → PCB en premier).
T2 Sophie voit le widget PCB en haut. À droite : #4 / 27 en jaune doré. Encadré ambre "Saison en cours · Printemps 2026", où elle est #3 sur 18 (depuis le début de la saison). Le widget liste "Mercredi 30 mai · 19:00 - Tournoi Hold'em" avec bouton S'inscrire.
T3 Elle clique S'inscrire. Le bouton se transforme en "Confirmer / Annuler". État local du composant. Pas encore d'appel réseau.
T4 Elle clique Confirmer. Le composant appelle la server action registerAsPlayerAction qui POST /api/players/me/clubs/poker-club-bordeaux/tournaments/<uuid>/register. Le backend crée une tournament_registration status='pending', player_id=Sophie.id. Réponse 201.
T5 Le bouton est remplacé par "En attente" en jaune doré. Le tournoi reste dans la liste avec son nouveau statut. L'admin du club voit la demande dans GET /api/club/tournaments/:id/registrations?status=pending.
T6 Quelques heures plus tard, l'admin valide. Sophie reçoit un email Resend. Au prochain refresh du dashboard, myRegistrationStatus devient "confirmed" (vert).

Données affichées dans un widget

  • En-tête : nom du club (lien vers la page club) + badge "Membre actif".
  • Rang global à droite : position dans le classement cumulé du club (groupé par nom comme le leaderboard public). Affiche aussi total points + nombre de tournois joués.
  • Encadré saison active (si une saison status='active' existe) : nom + date de fin + rang scopé saison.
  • 3 prochains tournois : nom + date + places libres + bouton/badge inscription.
  • Lien "Quitter ce club" en bas (server action leaveClubAction).

Edge cases

  • Joueur sans résultats dans le club : myRank=null → on affiche "Pas encore de résultats enregistrés dans ce club".
  • Aucune saison active : on omet simplement l'encadré ambre, pas de message d'erreur.
  • Aucun tournoi à venir : "Aucun tournoi à venir annoncé."
  • Tournoi members_only et joueur non membre actif → impossible normalement (les widgets ne sont rendus que pour les memberships actives), mais si le backend renvoie 403 members_only, le message d'erreur s'affiche sous le bouton.
  • Tournoi complet : spotsLeft=0, bouton désactivé.
  • Memberships pending/rejected : pas dans /dashboard, affichées via le composant simple <MembershipCard> en dessous des widgets.
